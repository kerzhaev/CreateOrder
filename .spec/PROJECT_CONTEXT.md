## 0. System Instructions (Agent Rules)
* **Encoding:** ВСЕГДА используй кодировку **Windows-1251** для всех файлов исходного кода VBA (`.bas`, `.cls`, `.frm`). Файлы `.md` и конфигурации хранятся в UTF-8.
* **VBA Standards:** Передавай пользовательские типы данных (UDT) только через **ByRef**.
* **UI/Forms:** Файлы форм (`.frm`) редактируются как текстовые документы. Бинарные файлы `.frx` запрещено изменять вручную — они должны оставаться нетронутыми.
* **Localization:** Все новые пользовательские русские строки для форм и диалогов должны проходить через `ModuleLocalization.bas` и лист книги `Localization`. Не размножать хардкод русских строк по формам и модулям без крайней необходимости.
* **Excel VBA Module Types:** Никогда не импортируй `ThisWorkbook` / `ЭтаКнига` и модули листов (`Лист1`, `Sheet1` и т.п.) как обычные `.bas`-модули через `VBComponents.Import`. Это **document/class modules**, а не standard modules. Их код нужно загружать только в уже существующий document-компонент через `CodeModule.DeleteLines/AddFromString` после нормализации текста без служебной шапки `VERSION 1.0 CLASS / BEGIN / Attribute VB_Name`.
* **Excel COM Hygiene:** Перед любыми изменениями `.xlsm` через Excel COM сначала проверяй висящие `EXCEL.EXE`. Фоновые зависшие процессы без окна могут silently ломать импорт модулей, формы и сохранение книги. Если процесс явно наш и без пользовательского окна, его нужно завершить до начала правок.
* **Workflow:** Мы следуем Spec-Driven Development. Любое изменение кода начинается с обновления или создания спецификации в папке `.spec/tasks/`.
* **Context Maintenance:** После выполнения задачи Агент обязан обновить разделы `Active State` и `Log` в этом файле.
* **Обновление документации:** При добавлении новой функции (кнопки на ленте, нового листа), Агент обязан также обновлять файл `docs/UserGuide.md`, добавляя описание новой функции для пользователей.

## 3.1. AI Factory / Pipeline Notes
* **Plan First:** Для новой фичи или заметного изменения сначала обязателен этап обсуждения/планирования, и только после согласованного плана разрешается реализация.
* **Branch First:** Перед началом новой фичи или нового самостоятельного change-set обязательно создавать отдельную рабочую ветку от актуального `main`. Не делать feature-реализацию прямо в `main`.
* **Merge Discipline:** Если изменения в ветке проходят проверку без ошибок, их нужно обязательно коммитить в этой ветке и только потом вливать обратно в `main`. Следующее изменение снова начинается с новой ветки.
* **Release Protection:** Не использовать `DPB -> DPx` как основной механизм `Unviewable Project`. На практике Excel показывает `contains invalid key 'DPx'`, предлагает recovery и затем все равно загружает проект. Текущая рекомендуемая схема релиза: штатный пароль на VBA-проект + `Ghost Modules`.
* **Build-Release:** `xl/vbaProject.bin` нельзя патчить через перекодировку всего файла в текст и обратно. Это повреждает OLE-стрим и дает repair/recovery-поведение. Патчинг релиза должен быть только byte-level и только по ASCII-сигнатурам фиксированной длины.
* **Ribbon Sync:** Недостаточно изменить только `resources/customUI14.xml` или только `.bas`-исходник. После добавления кнопки на ленту нужно синхронизировать и `customUI14.xml`, и callback-модуль (`mdlRibbonHandlers`) внутрь `CreateOrder.xlsm`, иначе кнопка видна, но не работает.
* **Workbook Verification:** После изменений в Ribbon или модулях, которые импортируются в книгу, проверять не только файлы в репозитории, но и содержимое самой `CreateOrder.xlsm`.
* **Document Modules:** `ThisWorkbook`/`ЭтаКнига` и worksheet-модули Excel нельзя переносить тем же пайплайном, что `bas/cls/frm`. Если экспортированный файл document-модуля загрузить как обычный модуль, Excel вставляет в код служебные строки `VERSION 1.0 CLASS`, `BEGIN`, `MultiUse = -1`, что ломает компиляцию (`Expected: end of statement`). Для document-модулей использовать только адресную запись кода в существующий компонент или `VbaModuleManager.importDocumentModule`.
* **Scripting.Dictionary:** Нельзя класть объектные ссылки в `Scripting.Dictionary` через `dict(key) = obj` для сценариев позднего связывания. Для объектов использовать `dict.Add key, obj`, иначе возможны runtime-ошибки `438 Object doesn't support this property or method`.
* **Validation Flow:** Если строка сотрудника однозначно разрешается хотя бы по одному идентификатору, валидация должна продолжать сбор всех доступных ошибок по строке. Полный early-exit допустим только для конфликтующих идентификаторов, дублей в `Штат` или полной невозможности разрешить сотрудника.
* **Safe Merge/Release:** Если рабочее дерево грязное и содержит посторонние изменения пользователя, merge в `main` и сборку релиза выполнять через отдельный чистый `git worktree`, чтобы не утянуть лишние файлы в push или в релиз.
* **Word Placeholder Limit:** Для Word COM нельзя полагаться на `.Find.Replacement.Text` для произвольных строк. При длинных значениях Word выбрасывает `Слишком длинный строковый параметр`. Длинные и потенциально длинные плейсхолдеры нужно вставлять через `Range`/`InsertAfter` кусками фиксированной длины.
* **Word Process Isolation:** Batch-экспорт не должен цепляться к уже открытому пользовательскому `Word.Application` и тем более не должен завершать все `WINWORD.EXE` через WMI. Для фоновой генерации документов использовать отдельный экземпляр Word и закрывать только его.
